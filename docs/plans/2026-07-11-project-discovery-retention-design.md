# 项目发现与留存闭环设计

日期：2026-07-11

## 目标

在保持 Roast / 测分为产品第一入口的前提下，将现有开发者、项目、组织、语言和关注数据组织成连续的站内探索旅程：用户从测分或榜单进入个人页后，可以自然进入项目、贡献者、相关项目与相似开发者，并在后续访问时继续上次的探索。

本轮只完成代码、测试和本地验证，不部署，也不写生产数据库。

## 已确认的现状

- `profile_snapshots` 已有 7,480 行，`developer_facets` 已有 25,151 行，原始数据充足。
- `repos` 与 `repo_developers` 当前均为 0 行，说明 Phase A/B 的代码已经存在，但历史项目图谱尚未回填。
- 项目详情沿用 `/developers/repo/{owner}/{name}`，避免新旧 URL 分裂 SEO。
- 首页已有 Roast、榜单预览与登录后的关注动态；个人页已有 Featured Work、Notable Contributions、相似开发者和 Battles。
- Phase A/B 当前仍是未提交工作区改动，本设计在其基础上继续实现。

## 方案选择

采用“分层探索架构”。

不采用只补搜索和 `/projects` 的轻量方案，因为它仍会让新页面彼此割裂；也不采用完整社交推荐方案，因为项目关注、行为事件库和推荐模型超出当前数据规模与本轮成本。

分层探索架构复用现有项目图谱、评分、站内热度和关注数据，新增项目内容流、全局搜索、上下文推荐和本地继续探索。它能形成闭环，同时避免引入新外部数据源或黑盒推荐。

## 信息架构

### 全局导航

Roast 保持第一项和首页主入口。Discover 改为分组菜单：

- Developers：现有开发者发现入口。
- Projects：新的 `/projects` 项目内容流。
- Languages：收敛后的语言分类入口。
- Organizations：收敛后的组织分类入口。

桌面端导航栏提供紧凑的全局搜索入口；移动端在菜单顶部提供搜索。搜索面板按开发者、项目和分类分组，支持键盘导航。

### `/projects` 内容流

`/projects` 使用项目卡片而不是标签墙。每张卡至少展示：

- 项目名称与一句话介绍。
- Star、主语言和 Topics。
- 贡献者数量、平均分和顶级贡献者。
- 综合质量和近期势头。
- 一条确定性推荐原因。

支持综合质量、近期势头和 Star 三种排序，支持语言筛选和稳定分页。首版不使用无限滚动，以保证返回定位、SEO 和可测试性。

### 页面下一跳

- 项目页：相关项目、同语言热门项目和值得认识的贡献者。
- 个人页：将相似开发者提前到 Featured Work 之后，并增加共同参与项目。
- 榜单页：增加按项目发现和按组织发现入口。
- 登录首页：保留关注动态，在测分区域后增加“继续探索”。
- 未登录首页：同样显示本地最近浏览，但不跨设备同步。

## 数据与排序

不新增外部数据源，复用以下数据：

- `repos`：项目身份、描述、Star、语言和 Topics。
- `repo_developers`：项目与 owner / contributor 关系。
- `scores`：贡献者评分、Tier 和影响力。
- `account_stats` 与近期 lookup：站内近期热度。

### 综合质量

综合质量使用可解释公式：

```text
quality = 平均贡献者分 × log2(贡献者数 + 1)
```

查询继续遵守 `hidden = 0` 和公开分数下限，避免隐藏账号或低质量数据进入推荐。

### 近期势头

近期势头聚合项目贡献者最近 7 天的站内 lookup 热度，并按项目规模归一化。它表达“站内近期关注”，不冒充 GitHub Star 增长，也不额外调用 GitHub。

### 推荐原因

推荐文案由确定性规则产生：

- GOD / ELITE 贡献者达到阈值：由多位顶级开发者维护。
- 近期热度进入前列：本周关注度上升。
- 平均分较高：贡献者质量领先。
- 与用户最近浏览语言一致：与你关注的某语言相关。

### 相关项目和共同项目

相关项目优先按共享贡献者数量排序，不足时使用相同语言和重叠 Topics 补齐。两个开发者的共同项目直接查询 `repo_developers` 交集，不引入向量数据库或 AI 推荐。

## 搜索

扩展现有 `/api/search-users`，保持 `{ users }` 向后兼容，同时返回项目和分类结果。项目优先按前缀匹配 owner/name 与项目名，再按 Star 和质量排序。

全局搜索复用同一个客户端组件和 API，不在 Navbar、首页 Omnibox 与移动菜单分别实现三套搜索逻辑。首页 Omnibox 保留 Roast 与 PK 意图解析，并合并项目结果。

## 继续探索

浏览历史保存在 `localStorage`：

- 最多 12 条。
- 只保存项目或开发者标识、显示信息和访问时间。
- 不保存搜索词。
- 不上传服务端，不跨设备同步。

继续探索组件读取最近项目和开发者，并与登录后的关注动态并列显示。存储不可用或历史为空时静默隐藏。

## 缓存与降级

- `/projects` 榜单与相关项目查询复用 Redis cache-aside，TTL 10 分钟。
- 项目搜索缓存 5 分钟。
- `repos` 为空时，项目发现显示明确空状态；现有项目详情继续降级为贡献者名单。
- 缺少描述、Topics、owner 或贡献者聚合时，项目卡只隐藏对应字段，不阻断页面。
- Redis 不可用时使用数据库查询或现有失败静默策略，不能导致主页面 500。
- 生产回填先 dry-run，再分页写入；本轮只在隔离的临时本地数据库执行。

## 本地数据验证

本地验证使用隔离数据库，不修改当前配置的远端数据库：

1. 从现有公开快照只读抽取一批代表性样本到临时本地库。
2. 对本地回填先运行 `dry=1`，核对 repo 与 link 数量。
3. 分页执行本地实际回填。
4. 验证 `repos`、`repo_developers` 数量和关系。
5. 抽查至少 3 个项目页和 3 个个人页，确认项目头卡和站内项目链接出现。

## 测试策略

所有新行为遵守 TDD：先写失败测试并确认失败，再写最小实现。

- 数据层：项目搜索、三种排序、分页、相关项目、共同项目、空数据与隐藏账号。
- API：旧响应兼容、新分组结果、空查询、数据库缺失和缓存降级。
- 组件：导航分组、搜索面板、项目卡、继续探索。
- 路由：`/projects` 筛选、排序、分页、metadata 和 canonical。
- 埋点：项目搜索、项目卡、推荐入口和继续探索点击。
- 回归：Phase A/B 现有 repo graph 和 DB 测试。

最终运行完整测试、`pnpm typecheck`、`pnpm lint` 和生产构建。启动本地站点后检查首页、Projects、项目页、个人页和榜单页，并按项目规范验证 Light、Dark、Auto、桌面和移动端。键盘搜索、焦点、空状态、返回导航和外链行为都纳入手工验收。

## 非目标

- 不部署，不执行生产回填。
- 不重新抓 GitHub，不重新评分或重新生成 Roast。
- 不新增项目关注、服务端浏览历史、跨设备同步、向量数据库或 AI 推荐。
- 不更换现有项目详情 canonical URL。
